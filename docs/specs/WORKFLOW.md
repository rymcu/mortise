# 规范驱动流程

本流程用于保障需求规范与实现代码同步版本化与评审。

## 1. 提案与草稿

- 新需求先创建规范文档 (参照 templates/feature-spec.md)
- 明确架构归属 (业务域/层级/模块), 避免跨层或同层依赖
- 规范 PR 标记为 Draft, 收集评审意见

## 2. 评审与批准

- 评审重点: 需求边界、接口设计、数据模型、兼容性、风险
- 评审通过后将状态更新为 Approved

## 3. 实现与追踪

- 实现 PR 必须引用已批准规范
- 规范文档里更新状态为 Implemented, 并补充变更记录

## 4. 变更与回滚

- 变更必须先更新规范, 再修改代码
- 重要变更需要补充迁移计划与回滚方案

## 5. 验收

- 验收标准必须可验证 (测试用例或验收清单)
- 验收完成后在规范中记录
