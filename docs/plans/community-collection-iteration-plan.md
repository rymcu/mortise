# 社区合集功能迭代计划

## 目标

为社区模块的“合集/专辑”能力建立明确的版本边界，先保证第一版可用、可发布，再在后续版本逐步补齐协作与编排能力，避免当前版本范围继续膨胀。

## 当前已完成范围

### 已落地能力

- 合集领域模型已建立：`Collection`、`CollectionArticle`、`CollectionMember`
- 数据库迁移已完成：新增合集、合集文章关联、合集成员关联三张表
- 公开接口已具备基础能力：合集列表、合集详情、我的可维护合集、创建/编辑合集
- 文章发布链路已支持绑定合集、专题、标签
- 社区前端已增加：
  - 专题列表页
  - 合集列表页
  - 合集详情页
  - 合集创建页
- 文章编辑器已改为左右分栏布局：左侧维护标题、封面、摘要、专题、标签、合集等元数据，右侧维护正文
- 评论编辑框已补充基础 Markdown 工具栏

### 当前版本定位

当前版本定位为 **V1 基础可用版**，目标是先打通：

- 用户可以创建合集
- 用户可以在发文时把文章加入合集
- 用户可以浏览专题与合集
- 用户可以在更完整的编辑界面中完成文章与评论创作

## 第一版范围冻结

### 第一版明确不增加的功能

以下能力原则上不进入第一版基础闭环；若仅为收口现有能力缺口，则允许在“受保护编辑页 / 创作中心”内补齐最小可用写侧，不向公开浏览链路扩散：

1. 合集邀请确认流、通知联动、审批机制
2. 面向公开详情页的创作者操作入口扩散
3. 文章详情页“所属合集/相关合集”展示区块
4. 面向运营后台的合集管理页面与高级审核能力
5. 合集统计分析、推荐位、热度排序等运营增强能力

### 第一版控制原则

- 只保留“基础可创建、可绑定、可浏览”的闭环
- 不增加新的交互链路，避免引入额外权限复杂度
- 不扩展新的后台运营流程，避免影响当前交付节奏
- 不为协作能力补充邀请制、审批制、消息通知等衍生机制

## 后续版本规划

### 第二版：合集协作能力

目标：让合集从“个人系列内容容器”升级为“多人共创容器”。

计划内容：

- 增加合集成员管理接口
- 支持 `owner / editor / contributor / viewer` 角色管理
- 支持添加成员、移除成员、调整角色
- 在合集详情页或设置页展示协作者列表
- 明确协作权限边界：
  - `owner`：全部权限
  - `editor`：编辑合集与维护文章顺序
  - `contributor`：向合集补充文章
  - `viewer`：仅查看成员可见合集

实施前置条件：

- 明确邀请方式是“直接添加”还是“邀请确认”
- 明确是否需要通知中心联动
- 明确成员可见合集的前端展示入口

### 第三版：合集编排能力

目标：让合集具备“系列课程/专栏”形态，而不只是文章聚合。

计划内容：

- 增加合集文章排序维护界面
- 支持拖拽排序或显式顺序号调整
- 支持维护 `note` 字段，用于说明每篇文章在合集中的定位
- 在合集详情页突出“阅读顺序”与“系列结构”
- 支持文章详情页展示所属合集与上下篇导航

实施前置条件：

- 明确排序交互形式
- 明确合集详情页的信息层级与视觉优先级
- 评估是否需要同步支持目录导航与系列进度标识

### 第四版：运营增强能力

目标：让合集进入可运营、可推荐、可分析阶段。

计划内容：

- 后台合集管理与状态控制
- 合集推荐位、精选合集、热门合集
- 合集维度统计指标：浏览量、订阅量、完读率等
- 与通知、活动、推荐算法联动

## 版本建议

### V1 发布建议

V1 仅验收以下能力：

- 合集基础模型与数据表稳定
- 合集基础接口可用
- 前端列表/详情/创建流程可用
- 发文绑定合集流程可用
- 文章编辑器与评论编辑器体验达到可用标准

## 基于当前代码的实现盘点（2026-03）

### 已实现

- 合集列表、详情、创建、我的可维护合集接口已具备
- 文章发布 / 编辑链路已支持绑定合集
- 合集详情页已展示协作者、文章顺序与备注读侧信息
- 管理后台已经存在**基础合集管理能力**
  - 分页查询
  - 状态切换
  - 删除 / 批量删除

### 部分完成

1. **合集编辑闭环不完整**
   - 后端 `createOrUpdate` 已支持更新
   - 前端仍只有“创建合集”页，缺少独立编辑页

2. **协作成员模型已落地，但缺少管理链路**
   - 已有 `CollectionMember`、角色枚举、owner 初始化逻辑
   - 缺少添加成员 / 移除成员 / 调整角色的 API 和前端界面

3. **排序与备注只有读侧，没有写侧**
   - `sortNo`、`note` 字段与详情页展示已存在
   - 缺少排序维护与备注编辑接口
   - 当前 `sortNo` 写入逻辑仍不是“合集内真实顺序”

4. **公开列表的状态 / 可见性过滤仍需收口**
   - 当前 V1 虽可浏览，但公开查询口径还需与访问策略统一

### 与原计划不一致

1. 原文将“后台合集管理”列为 V1 不做，但当前仓库已经提前具备基础后台管理页面与接口。
2. 原文将详情联动完全视为后续计划，但当前详情页已经提前具备部分协作者 / 顺序 / 备注展示能力。
3. 当前真正未完成的重点，不是“是否有合集能力”，而是“合集写侧管理能力和权限边界尚未补齐”。

### V1 发布后优先级

建议按以下顺序推进：

1. 合集协作成员管理
2. 合集文章排序与备注维护
3. 文章详情页系列化展示
4. 后台运营能力

8. **扩展后台合集管理**

## 风险与说明

- 协作能力一旦上线，权限模型、成员变更审计、通知联动都会同步变复杂，不适合继续塞入当前版本
- 排序与备注编辑看似是小功能，但会显著增加前端交互和服务端约束校验成本
- 如果第一版继续扩展，容易把“可发布版本”拖成“半完成的大版本”

## 结论

当前已在“仅创作中心 / 受保护编辑页可写、公开详情页不暴露编辑入口”的前提下补齐成员管理、文章编排写侧与文章详情页系列联动。

后续如需继续推进，以本计划为基线，按“运营能力增强”的顺序逐步实现。
