# 社区合集功能迭代计划

## 目标

为社区模块的“合集/专辑”能力建立明确的版本边界，先保证第一版可用、可发布，再在后续版本逐步补齐协作与编排能力，避免当前版本范围继续膨胀。

## 当前已完成范围

### 已落地能力

- 合集领域模型已建立：`Collection`、`CollectionArticle`、`CollectionMember`
- 数据库迁移已完成：新增合集、合集文章关联、合集成员关联三张表
- 公开接口已具备基础能力：合集列表、合集详情、我的可维护合集、创建/编辑合集
- 文章发布链路已支持绑定合集、专题、标签
- 社区前端已增加：
  - 专题列表页
  - 合集列表页
  - 合集详情页
  - 合集创建页
- 文章编辑器已改为左右分栏布局：左侧维护标题、封面、摘要、专题、标签、合集等元数据，右侧维护正文
- 评论编辑框已补充基础 Markdown 工具栏

### 当前版本定位

当前版本定位为 **V1 基础可用版**，目标是先打通：

- 用户可以创建合集
- 用户可以在发文时把文章加入合集
- 用户可以浏览专题与合集
- 用户可以在更完整的编辑界面中完成文章与评论创作

## 第一版范围冻结

### 第一版明确不增加的功能

以下能力不进入当前第一版实现范围，只记录计划，不在本轮继续开发：

1. 合集协作成员管理界面
2. 合集邀请、移除成员、角色调整流程
3. 合集文章排序维护界面
4. 合集文章备注（`note`）编辑界面
5. 文章详情页“所属合集/相关合集”展示区块
6. 面向运营后台的合集管理页面与高级审核能力
7. 合集统计分析、推荐位、热度排序等运营增强能力

### 第一版控制原则

- 只保留“基础可创建、可绑定、可浏览”的闭环
- 不增加新的交互链路，避免引入额外权限复杂度
- 不扩展新的后台运营流程，避免影响当前交付节奏
- 不为协作能力补充邀请制、审批制、消息通知等衍生机制

## 后续版本规划

### 第二版：合集协作能力

目标：让合集从“个人系列内容容器”升级为“多人共创容器”。

计划内容：

- 增加合集成员管理接口
- 支持 `owner / editor / contributor / viewer` 角色管理
- 支持添加成员、移除成员、调整角色
- 在合集详情页或设置页展示协作者列表
- 明确协作权限边界：
  - `owner`：全部权限
  - `editor`：编辑合集与维护文章顺序
  - `contributor`：向合集补充文章
  - `viewer`：仅查看成员可见合集

实施前置条件：

- 明确邀请方式是“直接添加”还是“邀请确认”
- 明确是否需要通知中心联动
- 明确成员可见合集的前端展示入口

### 第三版：合集编排能力

目标：让合集具备“系列课程/专栏”形态，而不只是文章聚合。

计划内容：

- 增加合集文章排序维护界面
- 支持拖拽排序或显式顺序号调整
- 支持维护 `note` 字段，用于说明每篇文章在合集中的定位
- 在合集详情页突出“阅读顺序”与“系列结构”
- 支持文章详情页展示所属合集与上下篇导航

实施前置条件：

- 明确排序交互形式
- 明确合集详情页的信息层级与视觉优先级
- 评估是否需要同步支持目录导航与系列进度标识

### 第四版：运营增强能力

目标：让合集进入可运营、可推荐、可分析阶段。

计划内容：

- 后台合集管理与状态控制
- 合集推荐位、精选合集、热门合集
- 合集维度统计指标：浏览量、订阅量、完读率等
- 与通知、活动、推荐算法联动

## 版本建议

### V1 发布建议

V1 仅验收以下能力：

- 合集基础模型与数据表稳定
- 合集基础接口可用
- 前端列表/详情/创建流程可用
- 发文绑定合集流程可用
- 文章编辑器与评论编辑器体验达到可用标准

### V1 发布后优先级

建议按以下顺序推进：

1. 合集协作成员管理
2. 合集文章排序与备注维护
3. 文章详情页系列化展示
4. 后台运营能力

## 风险与说明

- 协作能力一旦上线，权限模型、成员变更审计、通知联动都会同步变复杂，不适合继续塞入当前版本
- 排序与备注编辑看似是小功能，但会显著增加前端交互和服务端约束校验成本
- 如果第一版继续扩展，容易把“可发布版本”拖成“半完成的大版本”

## 结论

合集功能第一版到当前范围即可收口，不再继续增加协作管理、排序维护、详情联动等扩展能力。

后续如需继续推进，以本计划为基线，按“协作能力 → 编排能力 → 运营能力”的顺序逐步实现。
